# リポジトリワークフロールール

## ハンドブック優先（Handbook-first）

やり方を変えるときは **先にハンドブックを直して** から現場で実行する。

- このハンドブックが「今のやり方の正解」（唯一の場所）
- 現場だけ変えてドキュメントを放置しない
- 同じルールを複数の場所に書かない（SSoT）

## 変更の流れ

1. 変えたいことを整理（今 → 新 → 理由 → いつから）
2. ハンドブックのファイルを編集
3. PR を作成
4. レビューを受ける
5. マージ後、現場に伝える

## PR の書き方

PRには以下を書く:

- **何を直した**: 変更内容
- **どうして**: 変更理由
- **いつから**: 適用開始日
- **影響**: 関係する人や場所

## 確認手順

PRを出す前に:

```bash
mkdocs build    # ビルドが通ること
```

## 完了の定義

- [ ] `mkdocs build` が成功する
- [ ] 壊れたリンクがない
- [ ] nav に追加が必要なら mkdocs.yml を更新した
- [ ] 見た目（CSS/テーマ）を変えていない
- [ ] frontmatter を書いた
- [ ] last_checked を更新した

## 例外

- 誤字脱字の修正はレビューなしでマージしてよい
- 緊急の変更は先に口頭で伝えて、あとからPRを出す
